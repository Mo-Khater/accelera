\begin{abstract}
This report presents the individual design, implementation, verification, and learning journey behind my contributions to three important parts of the \systemname{} framework: the Pipeline Module, the Benchmark Module, and the Deployment Module. My work focused on making \systemname{} more complete as an engineering system, not only as a collection of machine-learning algorithms. The contributions described in this report cover internal execution behavior, user-facing platform features, authentication-related workflows, dashboard visibility, and the final deployment stage that transforms trained models into usable services.

Within the Pipeline Module, I contributed to the C++ back-end execution engine supporting branching workflows, topological sorting, DAG constraints, and duplicate-node optimization. In particular, the \modulepath{runParallel} functionality was a shared contribution between Mohamed Ashraf and me, focusing on thread-safe graph execution, dependency-aware branch scheduling, counting semaphores, and controlled memory release. Alongside this shared parallel-execution work, I also implemented a native Merge Node supporting class and probability-based majority voting strategies.

Within the Benchmark Module, I developed a Node.js REST API using Express and MongoDB to handle secure user authentication (JWT and bcrypt), dataset uploads, leaderboard management, and validation of submitted predictions against hidden ground truths. I also built a React frontend consisting of dedicated User and Admin Dashboards for task filtering, leaderboard viewing, and administrator-level user management.

Finally, the Deployment Module forms the main contribution of this individual work. It integrates a production-ready FastAPI web server featuring runtime data validation via Great Expectations, a request tracking and telemetry logging layer, a lightweight model Version Control System (VCS) with SHA-1 hash commits and rollback support, and an automated deployment engine targeting local Docker, Heroku, and remote AWS EC2 instances over SSH. This module was the most technically demanding part of my work because it required connecting machine-learning artifacts, configuration files, server-side APIs, Docker containers, command-line workflows, and remote deployment environments into one coherent system.
\end{abstract}

Developing and deploying machine learning pipelines in professional settings requires solving challenges that span different parts of the software lifecycle. A simple, linear script is often sufficient for training a single model on a local workstation. My individual work was motivated by this gap between a working experiment and a complete usable system. I wanted my contribution to help \systemname{} feel less like a narrow model-building prototype and more like a practical platform. \section{Machine Learning Pipeline Abstractions}
Machine learning pipelines are commonly used to organize workflows. The most popular pipeline system is the linear model in \modulepath{scikit-learn}, which chains transformers and estimators together. However, a linear pipeline cannot naturally represent complex, non-linear workflows. For example, if a developer wants to compare three scaling methods and two classifiers, a linear system requires building six separate pipelines. This duplicates both code and execution time, as early steps must run repeatedly. Modern systems address this by representing workflows as Directed Acyclic Graphs (DAGs), where nodes represent computational steps and edges represent data dependencies.

\section{Automatic Concurrency and GIL Release}
In Python-based frameworks, running pipelines concurrently is limited by the Global Interpreter Lock (GIL). The GIL prevents multiple threads from executing Python bytecodes at the same time. To achieve true parallel execution on multi-core systems, heavy computations must be offloaded to compiled native languages like C++ using binding tools like \modulepath{pybind11}. When executing C++ code, the Python GIL can be released (\modulepath{py::gil\_scoped\_release}), allowing separate native threads to run independently on different CPU cores.

\section{Web Serving and Data Quality Layers}
Model serving requires turning saved binaries (such as pickle files) into active web endpoints. FastAPI is a common choice for this, utilizing asynchronous loops to handle concurrent requests. However, raw inputs submitted by users can contain invalid formats or values that cause models to crash. Data validation tools like Great Expectations help prevent this. They define schema rules (such as expected columns, types, and range boundaries) to check data quality at runtime before sending inputs to the model.

\section{Multi-Target Deployment Platforms}
Deploying serving containers involves different target architectures:
\begin{itemize}[leftmargin=*]
    \item \textbf{Local Docker:} Provides local isolation, ensuring the service runs identically in development and production.
    \item \textbf{Heroku Container Registry:} Simplifies cloud hosting by managing HTTP routing and container scaling automatically.
    \item \textbf{AWS EC2:} Offers raw virtual machines in the cloud, requiring custom SSH management, codebase synchronization, remote container building, and manual firewall configuration.
\end{itemize}

\section{Related Deployment and Benchmarking Platforms}
Several public or paid platforms overlap with parts of my work. Heroku, Render, Railway, Google Cloud Run, and AWS EC2 can host web services or containers. Hugging Face Spaces can host model demos. Kaggle and similar competition platforms provide leaderboards and dataset-based comparison. These tools are useful, but they usually solve only one layer of the workflow.

The Deployment module in \systemname{} has a different role. It is not a general cloud provider. Instead, it prepares the machine-learning service itself: it loads model artifacts, reads \modulepath{config.json}, validates request payloads, exposes FastAPI endpoints, logs telemetry, creates deployment snapshots, and then connects the prepared service to local Docker, Heroku, or EC2. In other words, my module complements hosting platforms by building the ML-serving package that those platforms can run.

\section{Pipeline Module Architecture}
The Pipeline Module represents machine learning workflows as a DAG. Nodes are split into six functional categories: input, preprocessing, model, prediction, metric, and merge. The DAG is compiled by sorting the nodes topologically, which ensures that parent nodes execute before their child nodes.

This module was important to my work because it required understanding a machine-learning workflow as a connected structure rather than a sequential script. In a simple script, the programmer writes the order directly. In a graph system, the engine must infer a valid order from node connections, preserve dependencies, and make sure that data produced by one node is delivered correctly to the next node. Working on this part helped me understand that pipeline design is not only about adding model or preprocessing functions; it is also about controlling how information moves through the system.

Two major optimizations were introduced to improve execution performance. The duplicate-node sharing work is presented as part of my pipeline contribution, while the \modulepath{runParallel} execution functionality was developed as a shared contribution by Mohamed Ashraf and me:
\begin{enumerate}
    \item \textbf{Duplicate first-node sharing:} When multiple branches start with the same preprocessing step, the engine automatically shares the output of the first node rather than duplicating the calculation (Figures~\ref{fig:before_sharing} and \ref{fig:after_sharing}).
    \item \textbf{Active consumer counting:} The engine counts the active downstream consumers for each intermediate array. Once all child nodes finish execution, the parent node's temporary data is freed from RAM.
\end{enumerate}

\subsection{Merge Node Role in the Pipeline}
\noindent The Merge Node is one of the features that makes the graph pipeline more flexible than a simple linear pipeline. In a branch-based workflow, several paths may produce different predictions or probability outputs. Without a merge operation, the user would need to collect these outputs manually and write separate code to combine them. The Merge Node gives the graph a controlled point where multiple outputs can be joined into one final result.

From my perspective, implementing the Merge Node was challenging because it required understanding how upstream nodes communicate with downstream nodes. The output shape may differ depending on whether the branch produces direct class labels or probability vectors. Therefore, the merge behavior had to handle both direct predictions and probability-based predictions in a way that remained deterministic and understandable. This work taught me that graph nodes are not isolated functions; each node must respect the data contract between previous and next stages.

\subsection{Run Parallel Role in the Pipeline}
\noindent The \modulepath{runParallel} functionality allows independent graph branches to execute concurrently when their dependencies are satisfied. This improves the execution capability of \systemname{} because branch-heavy workflows often contain paths that do not depend on each other after a shared preprocessing stage. Running those branches sequentially wastes available CPU resources, while running them in parallel can reduce waiting time and improve the user experience for larger experiments.

The main difficulty in this work was thinking beyond sequential execution. A graph cannot simply start all nodes at once. Each node must wait until its parent nodes have finished, and shared intermediate data must remain available until all consumers are done. For this reason, the parallel scheduler needed synchronization, dependency checking, active consumer counting, and careful GIL handling. This was one of the parts of the project that made me think like a systems engineer rather than only as a machine-learning user.

\section{Pipeline Module Parallel Scheduler}
The parallel scheduler in \systemname{} is implemented in C++ in \modulepath{src/core/graph.cpp}. This \modulepath{runParallel} functionality was a joint contribution by Mohamed Ashraf and me. When parallel execution is enabled, the graph uses \modulepath{std::async} to execute independent nodes concurrently. The scheduling logic uses counting semaphores to limit the number of active CPU threads, and a binary semaphore to serialize GPU access.

This shared work was important because graph execution cannot be parallelized by simply launching every node at the same time. The scheduler must preserve dependency order, wait until parent nodes finish, protect Python object access through GIL handling, and release intermediate data only after all downstream consumers have finished using it. My role in this work focused on understanding how graph branches should be scheduled safely and how memory should be released without breaking later nodes, while Mohamed Ashraf also contributed to the design and implementation of the run-parallel execution flow.

Algorithm~\ref{algo:run_parallel} shows the parallel execution logic. It tracks finished and started nodes, releases the GIL during execution, and catches exceptions thrown by separate threads.

\begin{lstlisting}[style=algostyle,caption={Parallel Execution Engine in C++},label={algo:run_parallel}]
Input: execution_order nodes, cpu_threads limit
Output: executed graph with updated node states

1:   Release GIL via py::gil_scoped_release
2:   Initialize finished_executing map to false for all nodes
3:   Initialize started_executing map to false for all nodes
4:   Initialize CPU semaphore with cpu_threads count
5:   Initialize GPU semaphore with 1 (binary semaphore)
6:   Initialize atomic counter rem_nodes = size(execution_order)
7:   while rem_nodes > 0 do
8:       Acquire nodes semaphore
9:       for each node in execution_order do
10:          if started_executing[node] == true then continue
11:          check if all parent nodes are finished_executing
12:          if parents_finished == true then
13:              if node.uses_gpu == true then
14:                  Acquire GPU semaphore
15:              else
16:                  Acquire CPU semaphore
17:              end if
18:              set started_executing[node] = true
19:              spawn std::async(launch::async, [node]() {
20:                  Acquire GIL via py::gil_scoped_acquire
21:                  try {
22:                      node->execute()
23:                  } catch (...) {
24:                      store_exception()
25:                  }
26:                  decrement consumer counts for parent nodes
27:                  if parent consumer count == 0 then
28:                      release parent temporary memory
29:                  end if
30:                  set finished_executing[node] = true
31:                  decrement rem_nodes atomically
32:                  Release CPU/GPU semaphores
33:                  Release nodes semaphore
34:              })
35:          end if
36:      end for
37:  end while
38:  Re-throw any caught exceptions to the Python runtime
\end{lstlisting}

\subsection{Execution Flow and GIL Release Implementation}
The primary implementation challenge in the parallel execution flow was GIL (Global Interpreter Lock) contention. Because Python objects (such as scikit-learn estimators) are executed inside separate worker threads, concurrent calls will block each other if they attempt to manipulate Python references simultaneously. As part of the shared \modulepath{runParallel} contribution, Mohamed Ashraf and I structured a two-tier block wrapper in \modulepath{src/core/graph.cpp}:
\begin{enumerate}
    \item Before entering the loop-scheduling phase, the main thread releases the GIL using \texttt{py::gil\_scoped\_release}. This allows other native OS threads to run without waiting.
    \item Within each spawned worker thread, before invoking the node's \texttt{execute()} function, the GIL is explicitly acquired using \texttt{py::gil\_scoped\_acquire}. As soon as the call exits, the scope ends, immediately freeing the GIL for other concurrent threads.
\end{enumerate}

\subsection{Concurrency Semaphores and GPU Serialization}
To control the allocation of hardware resources, the shared \modulepath{runParallel} implementation uses custom semaphores based on C++ synchronization primitives:
\begin{itemize}[leftmargin=*]
    \item \textbf{CPU Thread Limiter:} A counting semaphore initialized to \texttt{ACCELERA\_NUM\_THREADS} (or system cores minus one). Worker threads block on this semaphore before execution, preventing context-switching overhead from over-subscription.
    \item \textbf{GPU Access Lock:} A binary semaphore (mutex) used to serialize accesses to CUDA contexts. Because multiple threads submitting operations to the same GPU concurrently can cause driver crashes, any node with the \texttt{uses\_gpu} flag is forced to acquire this binary lock, serializing GPU execution while keeping CPU-bound preprocessing branches running in parallel.
\end{itemize}

\subsection{Personal Reflection on Parallel Execution}
\noindent The most valuable lesson from the \modulepath{runParallel} work was that performance improvements must not break correctness. It is easy to describe parallel execution as ``running things at the same time,'' but the real engineering problem is deciding what is allowed to run at the same time. A node can only start when every parent has produced valid output. A parent buffer can only be released when every child that needs it has finished. If either rule is broken, the graph may produce wrong results, crash, or lose intermediate data.

This work helped me understand synchronization as a design responsibility. Semaphores, atomic counters, and maps for started and finished nodes were not only low-level programming tools; they were the mechanism that protected the graph's logical meaning. The experience also showed me why debugging parallel systems is difficult. A sequential bug often appears in the same place every time, while a parallel bug may depend on timing, scheduling, and the order in which threads finish. This made careful design and clear testing essential.

\section{Merge Node and Voting Strategies}
The \modulepath{MergeNode} class combines outputs from multiple prediction branches. Mazen implemented this node in C++ (\modulepath{src/nodes/merge.cpp}) and connected it to NumPy and SciPy components. The node supports a hard-voting strategy implemented in C++:
\begin{itemize}[leftmargin=*]
    \item \textbf{Direct Class Predictions (1D Arrays):} Collects predicted category labels from all upstream branches, stacks them into a single 2D NumPy array (rows correspond to samples, columns to model outputs), and uses \modulepath{scipy.stats.mode} to extract the majority class label.
    \item \textbf{Probability Predictions (3D Arrays):} Stacks predicted probabilities across all branches. It computes class labels for each model using \texttt{np.argmax} across the probability dimensions and then applies majority voting on the resulting class labels.
\end{itemize}

\subsection{Voting Strategy Implementation and Edge Cases}
In \modulepath{src/nodes/merge.cpp}, Mazen handled several implementation edge cases:
\begin{enumerate}
    \item \textbf{Single-Model Fallback:} If the merge node only receives a single input branch, it directly routes the input to the output without calling voting functions, avoiding unnecessary computation.
    \item \textbf{Tie-Breaking:} When the voting results are split evenly, the node falls back to selecting the output of the first model (the primary branch), guaranteeing deterministic results.
    \item \textbf{GIL Safety in Numpy Stack:} Because stack operations call back into the Python/NumPy interpreter, the merge execution block is wrapped in a GIL acquisition scope, securing memory allocations.
\end{enumerate}

The Merge Node was valuable because it made branch comparison more natural. Instead of forcing the user to leave the graph and manually combine model outputs, the graph itself can represent the final aggregation step. This makes the pipeline more expressive and closer to real experimentation, where different model branches may contribute to one final decision. Personally, this part improved my understanding of output shapes, deterministic behavior, and how a node should remain predictable even when receiving different kinds of upstream data.

\subsection{Runtime Schema Validation}
\noindent To prevent invalid inputs from causing model crashes or silent degradation, incoming payloads are validated against a schema specified in the configuration. The validation layer uses Great Expectations to construct an ephemeral data source context. Columns are validated for expected names, missing entries, unexpected inputs, and value bounds (e.g., minimum and maximum thresholds). Data types are coerced automatically (e.g., matching string, boolean, integer, or floating-point formats) to match the training format, raising a structured \texttt{SchemaValidationError} if coercion fails.

This was one of the most important design decisions in the module. A deployed model receives data from outside the training script, so it cannot assume that inputs are correct. The validation layer checks feature names, missing values, extra fields, numeric ranges, allowed values, and data types before inference. Personally, this part taught me that deployment quality is strongly connected to data quality. A model service should not only produce predictions; it should also reject invalid requests clearly.

Algorithm~\ref{algo:individual_config_validation} summarizes the validation workflow used before prediction.

\begin{lstlisting}[style=algostyle,caption={Deployment Configuration and Request Validation},label={algo:individual_config_validation}]

\noindent The version control subsystem is implemented in \modulepath{vcs.py}:
\begin{itemize}[leftmargin=*]
  \item \texttt{calculate\_hash(timestamp, message)}: Computes the 7-character SHA-1 hash value that acts as the commit identifier.
  \item \texttt{init(args)}: Configures the repository workspace by creating directories and saving the default index (\texttt{experiments.json}).
  \item \texttt{commit(args)}: Validates that models exist, generates a commit hash, copies the active configuration and models to \texttt{experiments/<hash>/}, and appends the commit entry to the registry.
  \item \texttt{log(args)}: Lists all recorded commits, showing details such as commit hash, creation date, message, and deploy status.
  \item \texttt{show(args)}: Displays the details and settings of a target commit.
  \item \texttt{deploy(args)}: Deploys a specific commit by copying its stored model files and configuration back into the active runtime workspace.
\end{itemize}

This figure is important because it shows that a deployment commit is more than a message: it captures the active configuration and model files together, then records them in the deployment registry so the state can be understood later.

The CLI was a deliberate design choice because deployment workflows are naturally command-based. Building an image, running a container, logging into Heroku, synchronizing files over SSH, and reading remote logs are all operations that developers usually perform from a terminal. My goal was to organize these operations into clear commands so that the user does not need to remember every low-level step.

  \item \textbf{local}: Orchestrates the complete local container pipeline by sequentially running the \textbf{prepare}, \textbf{build}, and \textbf{run-local} steps.
  \item \textbf{heroku-login}: Logs the client environment into the Heroku CLI and Heroku Docker registry.
  \item \textbf{heroku-deploy}: Packages and deploys the container to Heroku. It triggers login, pushes the Docker image to Heroku's registry, and releases the container for traffic.
  \item \textbf{heroku-push} and \textbf{heroku-release}: Granular command calls that push the container image and release it separately.
  \item \textbf{ec2-deploy}: Deploys to a remote AWS EC2 instance. It establishes an SSH connection, synchronizes files using \texttt{rsync}, compiles/builds the Docker container on the host, runs it, and probes local health targets.

\end{itemize}

\begin{itemize}[leftmargin=*]

  \item \textbf{commit}: Snapshots active model parameters and configurations under a message flag (\texttt{-m}).

\section{Personal Deployment Challenges and Decisions}
\noindent The Deployment module was the most personally demanding part of this work because it required connecting machine-learning artifacts with backend serving, validation, containers, and remote infrastructure. During implementation, I had to move beyond local model training and understand how a model becomes a long-running service that can receive HTTP requests, validate payloads, return predictions, and remain reproducible across different machines.

One major challenge was organizing the boundary between training artifacts and serving code. The solution was to make \texttt{config.json} the central contract between the training side and the deployed service. The service reads the configured model paths, loads all available preprocessors and the final estimator, and applies the same inference path for JSON and CSV requests.

Another challenge was learning how to make deployment repeatable. Instead of relying on manual copying and terminal commands, I implemented the orchestration script to prepare files, write runtime requirements, generate the Dockerfile, build the container, and support local, Heroku, and EC2 workflows. The EC2 path was especially useful as a learning exercise because it required SSH, \texttt{rsync}, remote Docker builds, container restart behavior, public ports, and health checks.

The VCS subsystem was an important design decision. Git already tracks source code, but the deployed model state also includes generated model files and runtime configuration. I therefore implemented a lightweight deployment-specific registry that snapshots \texttt{config.json} and the active model files, assigns a short hash, records metadata, and supports rollback by restoring a previous snapshot.

Testing the Deployment module also required special care. Many behaviors depend on Docker, Heroku, EC2, SSH, and file-system state, so the verification strategy used temporary workspaces and mocking to test the logic without depending on live cloud services. This made the module more reliable and helped separate the implementation logic from external infrastructure availability.

# Build independent model branches, then merge their predictions.

pipe.branch(

\section{Individual Challenges}
\noindent My individual work in \systemname{} required moving between different engineering layers. I worked on low-level graph execution behavior, user-facing benchmark interfaces, backend authentication flows, and deployment infrastructure. Each layer introduced a different kind of challenge, and the main difficulty was learning how to connect technical details to the overall value of the project.

\subsection{Pipeline Module Challenges}
\noindent The Pipeline Module challenged me to understand how node-based workflows behave internally. At first, it was easier to think of a pipeline as a sequence of steps. However, the graph module required a different mindset. Each node has parents, children, data dependencies, and execution rules. The Merge Node required me to understand how multiple upstream outputs can be combined without losing shape information or producing inconsistent results.

The run-parallel functionality was even more complex because it required thinking about execution order and synchronization. Independent branches can run at the same time, but only when their parent dependencies are complete. This made me realize that parallel execution is not only a performance feature; it is a correctness problem. If the scheduler starts a node too early or releases data too soon, the result may be wrong even if the code compiles. This challenge improved my ability to reason about graph execution and system state.

\subsection{Benchmark Module Challenges}
\noindent The Benchmark Module challenged me to think about usability. A backend route may return correct data, but the user still needs a clear interface to understand that data. The user dashboard had to present benchmarks and submissions in a way that felt useful rather than crowded. The admin dashboard had to show system-level information while remaining protected from normal users.

The authentication-related pages also required careful attention. Login and sign-up screens are simple visually, but they connect many parts of the system: user input, backend validation, password hashing, token creation, browser storage, navigation, and permission checks. I learned that a platform feels incomplete if the user flow is confusing, even when the backend logic works correctly.

\subsection{Deployment Module Challenges}
\noindent The Deployment module was the largest challenge because it required knowledge beyond ordinary machine-learning training. I had to understand configuration files, serialized model artifacts, serving APIs, Docker containers, SSH connections, remote servers, command execution, public ports, and health checks. At the beginning, these concepts felt separate. The main challenge was organizing them into one understandable workflow.

Debugging deployment-related behavior also required patience. A problem could come from a missing model path, an incorrect \texttt{config.json}, a Dockerfile issue, a port conflict, an unavailable remote host, or a public firewall setting. This taught me that deployment is not only about writing Python code. It is about making a system run correctly in an environment that has many external conditions. Because of this, I focused on clear commands, validation, generated files, and verification steps.

\section{Learned Technologies and Skills}

\section{Summary of Contributions}
In this graduation project, I designed, implemented, and verified key subsystems for \systemname{}:
\begin{itemize}[leftmargin=*]
    \item \textbf{Pipeline Execution Layer:} A shared contribution with Mohamed Ashraf to the \modulepath{runParallel} C++ scheduler utilizing semaphores for hardware thread limitation, safety gates for GPU tasks, and active consumer counting for memory release, paired with my NumPy/SciPy-based majority voting nodes.
    \item \textbf{Benchmark Subsystem:} A Node.js Express backend and MongoDB schema model supporting JWT/bcrypt authentication and a React frontend for task management, dashboards, and leaderboard display.
    \item \textbf{Serving and Deployment Infrastructure:} A production-ready FastAPI model server featuring Great Expectations schema validation, request tracking, model versioning (VCS) hash-based snapshotting, and automated scripts for local Docker, Heroku, and remote AWS EC2 deployment targets.
\end{itemize}

\section{Future Extensions}
Future improvements can focus on the following:
\begin{itemize}[leftmargin=*]
    \item \textbf{Horizontal Scheduling:} Extend the pipeline parallel scheduler to distribute graph branches across different cluster nodes instead of limiting execution to a single host.
    \item \textbf{Robust Encryption:} Improve data security in the benchmark module by encrypting ground truth files on MongoDB and verifying predictions inside sandbox runners.
    \item \textbf{Serverless Serving Targets:} Add serverless deployment capabilities (such as AWS Lambda) to the orchestrator to deploy lightweight models without maintaining active virtual servers.
\end{itemize}
