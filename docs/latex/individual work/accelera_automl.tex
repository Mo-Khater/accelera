\chapter*{Abstract}
\addcontentsline{toc}{chapter}{Abstract}
Selecting an appropriate machine-learning pipeline, including data preprocessing,
algorithm selection, and hyperparameter optimization, is a complex and
computationally expensive process. This thesis presents \systemname{}, an
automated machine-learning (AutoML) module integrated into the Accelera
framework. The proposed module provides a unified scikit-learn-style interface
for both classification and regression tasks while automating the construction
and optimization of machine-learning pipelines.

\medskip

\noindent
To efficiently explore the search space, \systemname{} combines conditional
configuration spaces, cross-validated model evaluation, multi-fidelity
optimization, and Bayesian optimization using a random-forest surrogate model
with an expected-improvement acquisition function. The system further
incorporates meta-learning warm starts, timeout-based isolation of candidate
evaluations, and ensemble construction through voting and stacking techniques.

\medskip

\noindent
This thesis describes the design, architecture, implementation, and evaluation
of \systemname{}, demonstrating how these components can be integrated into a
practical AutoML system that reduces manual effort while producing competitive
machine-learning models.

\chapter{Introduction}
\label{chap:introduction}

\systemname{} is an automated machine-learning module developed for the
Accelera framework. The project addresses the practical difficulty of selecting
and tuning machine-learning pipelines for tabular classification and regression
tasks. It provides a unified estimator interface while automating model
selection, hyperparameter optimization, fidelity scheduling, warm-start
selection, and ensemble construction.

\section{Motivation and Justification}

Machine-learning development is rarely limited to fitting one estimator. A
practical workflow requires the user to select a model family, define
its hyperparameters, decide whether features require scaling, choose an
evaluation metric, allocate computational resources, and compare alternatives
without leaking validation information. These decisions interact. For
example, a support-vector machine may benefit from feature scaling, while a
tree-based model usually does not require it. A configuration that performs
well on a small sample may fail when trained at full fidelity, and a highly
accurate model may be unsuitable if its search cost exceeds the available
budget.

\medskip

\noindent Automated machine learning (AutoML) treats these choices as a structured
optimization problem. Instead of manually testing a short list of models, an
AutoML system defines a search space and evaluates configurations under a
consistent protocol. The system must balance exploration of new regions with
exploitation of observations collected during the current search. It must also
handle practical failures such as invalid parameter combinations, estimator
exceptions, and trials that exceed their time limit.

\section{Project Objectives and Problem Definition}

The central problem addressed by this work is:

\begin{quote}
How can Accelera automatically produce a strong classification or regression
model under explicit trial and time constraints, while reducing unnecessary
full-cost evaluations and exposing a familiar estimator interface?
\end{quote}

\noindent This problem contains several subproblems:

\begin{enumerate}[leftmargin=*]
  \item representing diverse hyperparameter spaces;
  \item comparing configurations using a fair task-specific evaluation protocol;
  \item allocating computational resources across low and high-fidelity evaluations efficiently;
  \item transferring useful configurations from related historical datasets;
  \item selecting an effective final model or ensemble;
  \item isolating estimator failures so that individual candidates cannot terminate the entire search process.
\end{enumerate}

\section{Project Need}

Manual model selection is difficult to reproduce because it often depends on
informal trial-and-error decisions. It is also inefficient when many model
families have large hyperparameter spaces. A reusable AutoML component provides
three practical benefits. First, all candidates are evaluated by the same
metric and validation policy. Second, resource limits become explicit through
parameters such as \texttt{time\_budget}, \texttt{n\_trials}, and
\texttt{per\_run\_time\_limit}. Third, the complete run history can be exposed
as a leaderboard for inspection rather than returning an unexplained model.

\medskip
\noindent The module is implemented inside Accelera instead of as an isolated script so
that it can share the framework's packaging, preprocessing, reporting, and
future pipeline capabilities. The public API follows the fit/predict conventions
of scikit-learn, reducing the learning cost for users familiar with common
Python machine-learning libraries.

\section{Project Objectives}

The aim of this work is to design and implement a resource-aware AutoML engine
for tabular classification and regression. Its objectives are:

\begin{itemize}[leftmargin=*]
  \item provide \texttt{AutoMLClassifier} and \texttt{AutoMLRegressor};
  \item search multiple model families and their conditional parameters;
  \item support user-defined scoring, cross-validation, and reproducible seeds;
  \item evaluate candidates at progressively higher fidelity;
  \item use observed trials to guide later sampling;
  \item warm-start search using dataset metafeatures and stored historical runs;
  \item construct voting or stacked ensembles from competitive candidates;
  \item support parallel trials and per-trial timeout isolation;
  \item return the final model, best score, run history, and leaderboard.
\end{itemize}

\section{Project Outcomes}

The current implementation makes the following engineering and methodological
contributions:

\begin{enumerate}[leftmargin=*]
  \item A shared engine for classification and regression with task-specific
        evaluators and component registries.
  \item A three-stage fidelity schedule that varies sample fraction,
        cross-validation folds, and estimator budget.
  \item A sequential model-based optimizer using a random-forest surrogate and
        expected improvement over a sampled candidate pool.
  \item A promotion mechanism that advances successful low-fidelity candidates
        according to their relative cost.
  \item Classification and regression metafeatures for retrieving warm-start
        configurations from packaged metadata.
  \item Voting and stacked ensemble construction, including forward selection
        of diverse base models.
\end{enumerate}

\section{Scope}

This thesis focuses on supervised tabular classification and regression. The
implemented system assumes data that can be validated by scikit-learn's array
validation utilities. The work does not currently claim support for automated
deep neural architecture search, unsupervised learning, time-series-specific
validation, streaming updates, or distributed cluster execution. These are
possible extensions but are outside the present implementation scope.

\chapter{Market Visibility Study}
\label{chap:market}

This chapter discusses the market context for \systemname{} and explains why
an Accelera-native AutoML component is useful. It identifies the expected users
of the module, surveys related AutoML tools, and summarizes the practical value
of reducing repeated manual model-selection effort.

\section{Target Customers}

The target customers for \systemname{} are data scientists, machine-learning
engineers, researchers, and software developers who need a competitive tabular
classification or regression baseline without manually testing many estimator
families and hyperparameter settings.

\section{Market Survey}

Several AutoML tools address similar needs. Auto-sklearn combines Bayesian
optimization, meta-learning, and ensembles. FLAML emphasizes low-cost search
under limited resources. TPOT uses evolutionary pipeline search, while
AutoGluon Tabular focuses on strong stacked ensembles through a high-level
interface. \systemname{} is positioned as an Accelera-native alternative that
keeps the public estimator interface familiar while exposing internal search,
fidelity, and ensemble decisions for project-specific extension.

\section{Business Case and Financial Analysis}

The business value of \systemname{} comes from reducing the engineering time
required to establish reliable machine-learning baselines. Instead of assigning
a developer to manually compare preprocessing choices, models, and
hyperparameters, a project can run a bounded AutoML search and inspect the
resulting leaderboard. Operational costs are
limited because the module runs locally inside the user environment and does
not require a hosted service. Potential value increases when the same module is
reused across many Accelera projects, where each saved modeling cycle reduces
repeated manual effort.

\chapter{Literature Survey}
\label{chap:literature}

This chapter introduces the technical background needed to understand the
project. It covers combined algorithm selection and hyperparameter
optimization, search strategies, multi-fidelity evaluation, meta-learning,
ensemble learning, and related AutoML systems. The chapter ends by positioning
\systemname{} relative to existing tools and explaining the concepts used in
the implemented approach.

\section{Combined Algorithm Selection and Hyperparameter Optimization}

Given a set of algorithms $\mathcal{A}$ and a hyperparameter domain
$\Lambda_a$ for every algorithm $a$, combined algorithm selection and
hyperparameter optimization seeks

\begin{equation}
  (a^*, \lambda^*) =
  \arg\min_{a \in \mathcal{A},\,\lambda \in \Lambda_a}
  \widehat{L}(a, \lambda; D),
\end{equation}

\medskip
\noindent where $\widehat{L}$ is an estimated validation loss on dataset $D$. Auto-WEKA
formalized this combined view by searching over both learning algorithms and
their hyperparameters \cite{autoweka}. The domain is conditional: parameters
belonging to a random forest are inactive when the selected model is logistic
regression. This conditional structure motivates the use of the ConfigSpace
representation in \systemname{}.

\section{Search Strategies}

Grid search evaluates a fixed Cartesian product and scales poorly as the
number of dimensions increases. Random search allocates trials more effectively
when only a subset of hyperparameters strongly affects performance
\cite{randomsearch}. Sequential model-based optimization improves on uninformed
sampling by fitting a surrogate to observed configuration costs. SMAC uses a
random-forest model for algorithm configuration problems and selects candidates
through an acquisition function \cite{smac}.

\medskip
\noindent \systemname{} follows this general model-based pattern. It first evaluates
initial and warm-start configurations, represents each configuration as a
numeric vector, augments that vector with fidelity variables, and fits a
random-forest regressor. Predictions from individual trees provide an empirical
mean and standard deviation. Expected improvement then ranks a randomly sampled
candidate pool. The implementation is inspired by SMAC principles but is a
project-specific optimizer rather than a dependency on the external SMAC
package.

\section{Multi-Fidelity Evaluation}

Full cross-validation on an entire dataset is expensive, particularly for
large ensembles or boosting models. Multi-fidelity optimization first evaluates
candidates using cheaper approximations. Fidelity may be varied by the number
of training instances, the number of validation folds, training iterations, or
model capacity. Poor candidates can be discarded early, while promising ones
receive more resources.

\noindent In this work, fidelity is represented by a tuple

\begin{equation}
  f = (s, q, k, b),
\end{equation}

\noindent where $s$ is the stage index, $q$ is the sample fraction, $k$ is the number of
cross-validation folds, and $b$ is a multiplicative model budget. The fidelity
vector is included in the surrogate input so that observations from different
stages remain distinguishable.

\section{Meta-Learning for Warm Starts}

Meta-learning attempts to transfer knowledge across machine-learning tasks.
An AutoML system can characterize a new dataset using metafeatures, compare it
with previously evaluated datasets, and schedule successful historical
configurations early. Auto-sklearn combines Bayesian optimization,
meta-learning, and ensemble construction \cite{autosklearn}. The intended
benefit is not to guarantee that a historical configuration remains optimal,
but to reduce the cold-start period before useful observations are available.

\medskip

\noindent \systemname{} computes task-dependent statistical descriptors. Shared
descriptors include the number of instances and features, numeric/categorical
ratios, missing-value ratios, skewness, kurtosis, and categorical symbol
statistics. Classification additionally records class entropy, class-count
statistics, and class probabilities. Regression records target mean, standard
deviation, skewness, and kurtosis. Stored datasets are ranked by distance in
this metafeature space, and candidate configurations from the closest datasets
are converted back into active ConfigSpace objects.

\section{Ensemble Learning}

Ensembles combine several fitted models to reduce the risk of selecting one
unstable estimator. Voting uses the predictions or probabilities of multiple
base estimators. Stacked generalization instead trains a meta-model on
out-of-fold predictions from base models \cite{stacking}. Out-of-fold
generation is essential: fitting the meta-model on predictions made on each
base model's own training observations would produce leakage and an overly
optimistic result.

\medskip

\noindent The Accelera implementation provides both strategies. Stacking uses bagged
base estimators, creates out-of-fold prediction blocks, performs forward
selection, and fits a logistic-regression meta-model for classification or a
ridge-regression meta-model for regression. The original feature matrix may
optionally be appended to the prediction blocks.

\chapter{System Design and Architecture}
\label{chap:design}

The \systemname{} module was designed as a resource-aware AutoML workflow for
supervised tabular classification and regression. The design starts from a
simple user-facing estimator interface, then builds an internal search process
that validates the input data, constructs a conditional configuration space,
evaluates candidate pipelines, and selects a final fitted model. The
implementation was built around separate internal responsibilities for
configuration-space construction, candidate evaluation, optimization,
meta-learning warm starts, timeout handling, and ensemble selection so that each
part can be tested and extended without changing the public interface.

\section{Overview and Assumptions}

\systemname{} automates supervised tabular machine-learning pipeline selection
for classification and regression tasks. The module receives a dataset and user
configuration, searches over valid model and hyperparameter configurations,
evaluates candidates under resource limits, and returns a fitted final model
together with inspection outputs such as the leaderboard and run history.

\medskip

\noindent The primary user is a data scientist or developer who has a supervised
dataset and wants a competitive baseline without manually enumerating
estimators. A secondary user is a researcher who needs access to the run
history, leaderboard, fidelity stages, and ensemble decisions. A maintainer must
be able to add a model family without rewriting the optimization engine.

\medskip

\noindent The main use cases are:

\begin{itemize}[leftmargin=*]
  \item fit a classifier and obtain labels or probabilities;
  \item fit a regressor and obtain numeric predictions;
  \item restrict search to an allowed subset of model families;
  \item select a metric and cross-validation fold count;
  \item limit the complete search or each individual trial;
  \item enable or disable meta-learning and ensembles;
  \item inspect the best score, leaderboard, and complete run history;
\end{itemize}

\noindent The design assumes that the input data can be represented as tabular
feature and target arrays.

\section{System Architecture}

At a high level, \systemname{} follows a pipeline architecture. The user
interacts with a public estimator, the estimator validates the input data, the
AutoML engine builds a task-specific search space, and the optimizer repeatedly
selects candidate configurations for evaluation. The evaluator scores
candidates using cross-validation and returns structured results to the
optimizer. Meta-learning can provide warm-start configurations, and the ensemble
stage can combine strong candidates before final model selection.

\section{Accelera AutoML Module Design}

\subsection{Method Overview}

The complete method is organized as the following sequence:

\begin{enumerate}[leftmargin=*]
  \item validate the feature matrix and target;
  \item resolve compatible or user-allowed model families;
  \item construct the conditional configuration space;
  \item compute dataset metafeatures and retrieve warm-start configurations;
  \item evaluate initial configurations at low fidelity;
  \item fit a surrogate after sufficient finite observations exist;
  \item select new candidates by expected improvement;
  \item promote strong candidates through higher fidelity stages;
  \item fit the best full-fidelity individual configuration;
  \item optionally construct and evaluate an ensemble.
  \item return whichever fitted candidate has the stronger validation score.
\end{enumerate}

\subsection{Configuration-Space Construction}

The configuration space defines the set of machine-learning pipelines that the
AutoML engine is allowed to explore. In this work, classification and regression
use independent component registries because the available estimators and their
valid hyperparameters differ between task types. Each component in a registry
specifies four main elements: a model name, a hyperparameter factory, conditional
rules, and an estimator builder.
\\
The global configuration contains a categorical variable called
\texttt{model\_name}. This variable represents the learning algorithm selected
for a trial. After a model is selected, only the hyperparameters belonging to
that model are activated. Hyperparameters belonging to other models remain
inactive and are ignored during evaluation. This conditional structure prevents
the optimizer from treating irrelevant parameters as active decisions. For
example, the number of neighbors is meaningful for a k-nearest-neighbors model,
but it is not meaningful for a random forest or a support-vector machine.
\\
Formally, let the set of candidate models be

\[
\mathcal{M} = \{m_1, m_2, \ldots, m_k\}.
\]
\\
Each model \(m_i\) has its own hyperparameter space \(\Lambda_i\). The full
conditional configuration space can be written as

\[
\Lambda =
\bigcup_{i=1}^{k}
\left(
\{\texttt{model\_name} = m_i\} \times \Lambda_i
\right).
\]
\\
This formulation means that each valid configuration consists of two parts: the
selected model and the hyperparameters that are valid for that selected model.
Therefore, the optimizer searches only within the subspace associated with the
chosen estimator, instead of searching over all possible hyperparameters at the
same time.

\noindent Before constructing the final search space, the engine resolves the allowed model
set. The user may provide this set explicitly when only specific algorithms
should be considered. If no explicit set is provided, the engine starts from the
default registry for the task. The engine can then remove incompatible models
based on dataset properties. For example, some algorithms assume non-negative
feature values and can be excluded when the training data contains negative
values.

\noindent Restricting the search space has two main benefits. First, it reduces invalid
trials because the optimizer is less likely to sample configurations that cannot
be evaluated successfully. Second, it allows domain knowledge to be included
without changing the optimizer itself. The optimizer only receives the final
valid configuration space, while model compatibility and filtering rules are
handled by the AutoML engine. This separation makes the system easier to extend,
because new models can be added by registering their hyperparameters,
conditions, and estimator builder without rewriting the search procedure.

\subsection{Evaluation and Cost Conversion}

After a configuration is sampled, the evaluator is responsible for estimating
its predictive performance. For a configuration \(\lambda\), the engine first
builds the corresponding estimator and determines the preprocessing strategies
that are allowed for the selected model. The evaluator then tests each
model-preprocessor pair using cross-validation. Among all successful
preprocessing choices, the pair with the best validation performance is retained
as the evaluation result for \(\lambda\).

\noindent Let \(m(\lambda)\) denote the mean validation metric obtained by
cross-validation. Since different metrics have different optimization
directions, the evaluator converts the metric value into a cost value
\(c(\lambda)\). The optimizer always minimizes this cost, so smaller values
indicate better configurations.

\noindent For metrics that are naturally minimized, such as mean squared error or mean
absolute error, the cost can be used directly:

\[
c(\lambda) = m(\lambda).
\]

\noindent For metrics that are naturally maximized, such as accuracy, \(F_1\)-score, or
\(R^2\), the evaluator converts the score into a minimization objective. A common
conversion is

\[
c(\lambda) = 1 - m(\lambda),
\]

\noindent when the metric is bounded between 0 and 1. For metrics that are not strictly
bounded in this range, the evaluator may instead use a negated form:

\[
c(\lambda) = -m(\lambda).
\]

\noindent This conversion gives the optimizer a consistent objective regardless of the
original metric. In all cases, minimizing \(c(\lambda)\) corresponds to improving
the validation performance of the configuration.

\noindent If an evaluation fails, the evaluator does not allow the error to terminate the
complete AutoML search. Instead, the failed configuration receives a predefined
failure cost and the error message is stored in the run history. This allows the
optimizer to continue exploring other configurations while still recording which
trials were invalid or unsuccessful.

\noindent The validation procedure depends on the learning task. For classification,
stratified cross-validation is used so that each fold preserves the class
distribution as much as possible. Stratified subsampling is also used when only a
fraction of the training data is evaluated at a lower fidelity. This is important
because random subsampling may accidentally remove minority classes from small
subsets.

\noindent For regression, ordinary K-fold cross-validation is used because there are no
discrete class labels to preserve. Random subsampling is also sufficient for
regression tasks. In classification, the requested number of folds may be
adjusted when the smallest class does not contain enough samples to support the
requested fold count. For example, if the smallest class contains only three
samples, then using five stratified folds is not valid. In this case, the
classifier reduces the fold count to a feasible value. This prevents invalid
cross-validation splits and improves the robustness of the evaluation process.

\subsection{Three-Stage Fidelity Schedule}

To reduce the cost of the AutoML search, the engine evaluates configurations
using a three-stage fidelity schedule. Instead of evaluating every configuration
using the full dataset, the full number of cross-validation folds, and the full
model capacity, the system begins with a cheaper approximation. Only
configurations that perform well at cheaper stages are promoted to more
expensive stages.

\noindent Let \(s \in \{0,1,2\}\) denote the fidelity stage. Each stage is described by
three values: the sample fraction \(\rho_s\), the number of cross-validation
folds \(k_s\), and the model budget \(b_s\). Therefore, the trial specification
at stage \(s\) can be written as

\[
\tau_s = (\rho_s, k_s, b_s).
\]

\noindent The sample fraction controls how much of the training data is used during
evaluation. The number of folds controls the reliability of the validation
estimate. The model budget controls supported model-capacity parameters, such as
the number of estimators in ensemble models or the number of training iterations
in iterative models. Early stages are therefore cheaper because they use less
data, fewer folds, and smaller model budgets.

\noindent The evaluator scales supported capacity parameters using the model budget. If
\(\theta\) is a capacity parameter and \(\theta_{\max}\) is the value requested
by the sampled configuration, the stage-specific value can be written as

\[
\theta_s =
\max
\left(
\theta_{\min},
\left\lfloor b_s \theta_{\max} \right\rfloor
\right),
\]

\noindent where \(\theta_{\min}\) is the minimum valid value for that parameter. This keeps
the reduced model valid while still preserving the relative capacity requested
by the configuration.

\noindent A newly sampled configuration is first evaluated at stage zero. After a
successful evaluation, the optimizer stores the observed cost in a
per-configuration state. Once enough observations are available at a given
stage, the optimizer ranks configurations according to their costs. Since the
optimizer minimizes cost, smaller values are better. Only configurations inside
the configured top promotion quantile are advanced to the next stage.

\noindent The promotion rule can be written as

\[
\lambda \text{ is promoted from stage } s
\quad \Longleftrightarrow \quad
c_s(\lambda) \leq Q_q
\left(
\{c_s(\lambda') : \lambda' \in \mathcal{C}_s\}
\right),
\]

\noindent where \(c_s(\lambda)\) is the cost of configuration \(\lambda\) at stage \(s\),
\(\mathcal{C}_s\) is the set of configurations evaluated at stage \(s\), and
\(Q_q\) is the cost threshold for the configured promotion quantile.

\noindent The schedule spends the largest computational cost on configurations that were
already competitive under cheaper approximations. This avoids applying
full-fidelity evaluation to every sampled configuration and makes the search

\subsection{Surrogate-Guided Candidate Selection}

The search space used by the AutoML engine contains categorical,
continuous, integer, and conditional hyperparameters. Since the surrogate model
requires numerical input, every sampled ConfigSpace configuration is first
converted into a fixed-length numerical vector. Active hyperparameters are
encoded using their numerical or encoded categorical values. Missing or inactive
dimensions are replaced by finite sentinel values so that all configurations
have the same vector length.

\noindent Let \(x_{\lambda}\) denote the vector representation of configuration
\(\lambda\). Since the same configuration can be evaluated at different fidelity
levels, the vector is extended with the fidelity information. The augmented
input vector is defined as

\[
x' = [x_{\lambda}, s, q, k, b],
\]

\noindent where \(s\) is the fidelity stage, \(q\) is the sample fraction, \(k\) is the
number of cross-validation folds, and \(b\) is the model budget. Including these
values allows the surrogate model to distinguish between cheap low-fidelity
evaluations and expensive full-fidelity evaluations of the same configuration.

\noindent During the initial exploration phase, configurations are sampled without relying
on the surrogate model. This provides the optimizer with a set of observed
configuration--cost pairs. After enough successful evaluations have been
collected, the optimizer fits a random-forest regressor using only finite
observed costs. Failed evaluations or non-finite results are excluded from the
surrogate training data because they do not provide a reliable numerical target.

\noindent The random forest acts as an approximation of the unknown cost function. If
tree \(t\) predicts the cost \(\hat{c}_t(x)\) for candidate vector \(x\), then
the surrogate mean prediction is estimated as

\[
\mu(x) =
\frac{1}{T}
\sum_{t=1}^{T}
\hat{c}_t(x),
\]

\noindent where \(T\) is the number of trees in the forest. The uncertainty estimate is
computed from the disagreement between the trees:

\[
\sigma(x) =
\sqrt{
\frac{1}{T}
\sum_{t=1}^{T}
\left(
\hat{c}_t(x) - \mu(x)
\right)^2
}.
\]

\noindent A small value of \(\sigma(x)\) means that the trees give similar predictions,
while a larger value means that the trees disagree more strongly. Therefore,
\(\mu(x)\) estimates the expected cost of a candidate, and \(\sigma(x)\)
estimates the uncertainty of that prediction.

\noindent The optimizer then uses expected improvement to rank candidate configurations.
Because the objective is to minimize cost, improvement is measured relative to
the best observed cost \(c^+\). For a candidate \(x\), the predicted improvement
is

\[
I(x) = c^+ - \mu(x) - \xi,
\]

\noindent where \(\xi\) is a small exploration parameter. A larger value of \(\xi\)
encourages the optimizer to prefer candidates that are more likely to produce a
clear improvement rather than only a small predicted gain.

\noindent The standardized improvement is

\[
z(x) =
\frac{I(x)}{\sigma(x)}.
\]

\noindent The expected improvement is then computed as

\[
\operatorname{EI}(x) =
I(x)\Phi(z)
+
\sigma(x)\phi(z),
\]

\noindent where \(\Phi\) is the standard normal cumulative distribution function and
\(\phi\) is the standard normal probability density function. The first term
rewards candidates with a low predicted cost, while the second term rewards
candidates with higher predictive uncertainty. Therefore, expected improvement
balances exploitation and exploration.

\subsection{Warm-Start Selection}

Warm-start selection gives the optimizer an informed starting point by reusing
configurations that performed well on related historical datasets. For a new
dataset \(D\), the engine first computes a task-specific metafeature vector
\(m_D\). Historical datasets are represented by stored metafeature vectors
\(m_i\), together with their previously successful configurations.

\noindent The similarity between the new dataset and each historical dataset is measured
using a normalized distance. Let \(r\) be a scaling vector for the metafeature
dimensions. The distance between the current dataset \(D\) and a historical
dataset \(i\) is

\[
d(D,i) =
\left\|
\frac{m_D - m_i}{r}
\right\|_2 .
\]

\noindent The normalization prevents large-scale metafeatures, such as the number of
samples or features, from dominating smaller-scale metafeatures. Historical
datasets are then ranked by increasing distance, so datasets with smaller
distances are considered more similar to the current task.

\noindent After ranking the historical datasets, the engine gathers candidate
configurations from the nearest datasets. These configurations are not inserted
blindly into the search. First, configurations whose model names are not present
in the active model set are removed. This ensures that warm starts respect the
current user constraints and dataset compatibility checks. Next, configurations
are deduplicated using their model name and parameter signature, so repeated
historical suggestions do not occupy multiple positions in the initial queue.

\noindent Each remaining historical configuration is then converted into a valid
ConfigSpace configuration for the current search space. During this conversion,
invalid or unsupported parameters are skipped safely. This can happen, for
example, when the current version of a component does not contain a parameter
that existed in a stored historical configuration, or when the current active
space has stricter bounds. A failed conversion does not stop the search; it only
removes that warm-start candidate.

\noindent The number of historical datasets used, the number of configurations taken from

historical suggestions from dominating the search.

\noindent Warm starts are inserted into the initial configuration queue before uninformed
random samples. However, they are still evaluated on the current dataset using
the same validation and cost-conversion procedure as all other configurations.
Therefore, warm starts only guide the early search; they do not bypass
evaluation and cannot directly force the final selected model.

\subsection{Parallel Execution and Timeout Control}

The optimizer may suggest a batch of trials in one scheduling step. A batch can
contain promoted configurations from earlier fidelity stages, warm-start
configurations from historical datasets, and newly proposed configurations
selected either by the surrogate model or by random sampling. This batching
allows the search to use multiple CPU cores while keeping the optimizer logic
centralized in the parent process.

\noindent For parallel execution, each trial is executed in a separate worker process. The
worker receives the configuration, builds the corresponding estimator, evaluates
it, and returns an \texttt{EvaluationResult} through a process-safe queue. The
parent process does not wait passively for all workers to finish. Instead, it
periodically checks the worker queues, collects completed results, and monitors
the elapsed time of each running process.

\noindent If a worker process exceeds the configured
\texttt{per\_run\_time\_limit}, the parent process terminates it and records the
trial with a timeout status. The timeout result is stored in the run history in
the same way as other failed evaluations, together with a failure cost. This
ensures that one slow or non-terminating estimator does not block the complete
AutoML search.

\noindent Process-level execution is used instead of thread-level execution because some

fit procedure. Terminating a separate process is more robust than attempting to
stop a thread inside the same Python process. This is especially useful when an
estimator hangs, becomes extremely slow, or calls compiled code that does not
regularly return control to Python.

\noindent Overall, this design improves robustness and resource control. The optimizer can
continue receiving valid results even when some trials fail or time out, and the
caller can control the amount of parallel work performed by the search.


\subsection{Final Model and Ensemble Selection}

After the search phase finishes, the engine selects the strongest configuration
evaluated at full fidelity. This configuration is rebuilt from its stored model
name, preprocessing choice, and hyperparameters, and then fitted on all
available training data. This final refitting step is necessary because the
search phase estimates performance using validation folds, while the final model
should use the complete training set before being returned to the user.

\noindent If ensemble construction is disabled, the refitted best individual model is
returned directly. If ensembles are enabled, the engine considers successful
candidates from the run history as possible ensemble members. Failed trials,
timed-out trials, and configurations without valid fitted estimators are ignored.
This ensures that the ensemble is built only from models that were evaluated
successfully during the search.

\noindent The ensemble construction stage supports two strategies: voting and stacking.
For voting, the engine selects a bounded set of competitive estimators from the
run history. These estimators are combined using a task-specific voting model.
For classification, the voting ensemble combines class predictions or class
probabilities. For regression, it combines numeric predictions. Voting is simple
and robust because it does not train an additional meta-model.

\noindent Stacking uses a more expressive structure. Instead of directly averaging the
base-model outputs, it trains a meta-model that learns how to combine them. The
engine first considers candidate diversity before selecting base estimators, so
the stack is not formed only from many near-identical models. Each selected base
estimator is wrapped in bagging to improve stability. The base estimators then
produce out-of-fold predictions, which are used as training features for the
meta-model.

\noindent Forward selection is used to choose the final set of base estimators for the
stacking ensemble. The procedure starts with the strongest single candidate.
Additional candidates are tested one at a time. A candidate is kept if adding it
improves the validation score of the meta-model. The process stops when no
remaining candidate improves the score, or when the configured ensemble size is
reached. However, the minimum base-model requirement is also respected, so the
engine can include enough base estimators to form a valid stack even if early
improvements are small.

\noindent For classification, each base estimator may output a probability vector over the
class labels. Before these outputs are passed to the meta-model, the probability
columns are aligned according to the global class order. This avoids a common
problem where different classifiers store class labels in different orders. If
the columns were not aligned, the meta-model could interpret the probability of
one class as if it belonged to another class. After alignment, the probability
blocks from all base estimators are concatenated and passed to logistic
regression as the classification meta-model.

\noindent For regression, the stacking representation is simpler. Each base estimator
contributes one numeric prediction column. These columns are concatenated to
form the meta-feature matrix, and ridge regression is used as the regression
meta-model. Ridge regression is suitable here because it can combine correlated
base predictions while controlling the size of the learned coefficients.

\noindent The best individual model is always available as a safe fallback. The ensemble
is only returned when it performs at least as well as this individual model under
the same validation metric. The final decision can be written as

\[
\text{Final model} =
\begin{cases}
\text{ensemble}, & \text{if } S_{\text{ens}} \geq S_{\text{best}}, \\
\text{best individual model}, & \text{otherwise},
\end{cases}
\]

\noindent where \(S_{\text{ens}}\) is the validation score of the ensemble and
\(S_{\text{best}}\) is the validation score of the strongest individual model.
This rule prevents the system from returning a more complex ensemble unless it
provides at least the same predictive quality as the best single model. As a
result, ensemble construction can improve performance when the base models
complement each other, while still preserving the simpler individual model when
the ensemble does not provide a measurable benefit.

\subsection{Public Estimator Lifecycle}

\texttt{BaseAutoML} extends scikit-learn's \texttt{BaseEstimator}. Its
\texttt{fit} method validates $X$ and $y$, resets previously fitted state,
builds a task-specific engine, runs the search, and copies the result into
public fitted attributes. Prediction validates $X$ and delegates to the final
model. Calling prediction before fitting raises an explicit runtime error.

\medskip

\noindent The result state includes:

\begin{itemize}[leftmargin=*]
  \item \texttt{best\_model}: strongest fitted individual candidate;
  \item \texttt{ensemble}: fitted ensemble when constructed;
  \item \texttt{final\_model}: model selected for prediction;
  \item \texttt{best\_score}: validation score of the final choice;
  \item \texttt{leaderboard}: ranked model and ensemble summary; and
  \item \texttt{runhistory}: detailed trial records.
\end{itemize}

\subsection{Engine Collaboration}

The engine uses composition rather than embedding all behavior in one class.
It constructs a configuration space, evaluator, optimizer, and optional
ensemble. This separation allows the evaluator to remain task-aware while the
optimizer operates mainly on configurations, numeric vectors, and scalar costs.

\medskip

\noindent The runtime interaction is:

\begin{verbatim}
User estimator
  -> BaseAutoML.fit
  -> AutoMLEngine.search
     -> ConfigSpace construction
     -> metafeature extraction and warm starts
     -> task evaluator
     -> Optimizer.optimize
        -> staged trials / process workers
        -> surrogate and expected improvement
        -> promotion queue
     -> final individual model
     -> optional voting or stacked ensemble
  -> AutoMLResult
  -> fitted public estimator
\end{verbatim}

\subsection{Model Component Abstraction}

Classification and regression components encapsulate four related concerns:
their public name, hyperparameter definitions, conditional activation rules,
and estimator construction. This abstraction prevents the optimizer from
containing model-specific \texttt{if} statements. Adding an estimator requires
registering a component and ensuring its parameters can be scaled at the
defined fidelity levels when applicable.

\medskip

\noindent The current registries include estimators from scikit-learn and optional
gradient-boosting libraries. Because all models do not expose identical
interfaces, the component builders normalize values such as textual
representations of \texttt{None}, class weights, and task-specific objectives.

\subsection{Evaluation Result Model}

Every evaluation returns a structured record rather than only a score. The
record contains the model name, resolved parameters, preprocessing choice,
metric score, optimizer cost, duration, status, error text, stage, sample
fraction, fold count, and model budget. This representation serves three
purposes: it supports the optimizer, preserves failure diagnostics, and enables
leaderboard or experimental analysis without repeating the run.

\subsection{Meta-Learning Data Packaging}

Classification and regression metadata are stored in separate JSON
directories. File selection depends on task, metric, and classification target
shape. Paths are resolved relative to the installed module rather than a
developer-specific absolute directory. The package configuration explicitly
includes these JSON resources in built wheels so warm starts remain available
after installation.

\subsection{Ensemble Implementation Details}

The classification stacker includes an adapter that guarantees a probability
interface. If a base model lacks \texttt{predict\_proba}, decision scores are
converted to probabilities where possible; otherwise predicted classes are
encoded into probability-like output. Class columns are aligned so each base
block has consistent semantics.

\medskip

\noindent Both stackers generate out-of-fold predictions using cross-validation and fit
fresh bagged models on all training data for later inference. Forward-selection
state is stored as a fitted attribute together with its score. This explicit
state is also used by verbose structural logging and avoids recomputing the
selection result.

\subsection{User-Facing Example}

Listing~\ref{lst:usage} shows the intended public API.

\begin{lstlisting}[style=pythonstyle,caption={Classification with Accelera AutoML},label={lst:usage}]
from sklearn.datasets import make_classification
from sklearn.model_selection import train_test_split

from accelera.accelera_automl import AutoMLClassifier

X, y = make_classification(
    n_samples=1000,
    n_features=20,
    n_informative=12,
    random_state=42,
)
X_train, X_test, y_train, y_test = train_test_split(
    X, y, test_size=0.2, stratify=y, random_state=42
)

model = AutoMLClassifier(
    time_budget=300,
    n_trials=20,
    cv=3,
    random_state=42,
    search_n_parallel=2,
    stack_n_jobs=-1,
    use_meta_learning=True,
    use_ensemble=True,
    ensemble_strategy="stacked",
)
model.fit(X_train, y_train)

predictions = model.predict(X_test)
print(model.score(X_test, y_test))
print(model.return_leaderboard(top_n=3))
\end{lstlisting}

\chapter{System Testing and Verification}
\label{chap:testing}

This chapter explains how the correct operation and performance of
\systemname{} were tested and verified. It describes the benchmark setup,
testing strategy, comparative classification and regression results,
reproduction assets, and threats to validity. The goal is to show both that the
implementation works as intended and that its measured behavior can be
inspected and reproduced.

\section{Testing Setup}

The experiments compare \systemname{} with Auto-sklearn, FLAML, and TPOT on
supervised tabular classification and regression tasks. The evaluation asks
two practical questions: whether the systems attain comparable predictive
quality, and how much wall-clock time they require to do so. Classification is
evaluated primarily by accuracy, while regression is evaluated by the
coefficient of determination ($R^2$), root mean squared error (RMSE), and mean
absolute error (MAE). Runtime is reported in seconds for both task types.

\section{Testing Plan and Strategy}

The classification experiments were executed through the open-source
AutoMLBenchmark (AMLB) runner, available at
\url{https://github.com/openml/automlbenchmark.git}. They use the
\modulepath{flaml\_paper\_39} classification suite and the
\modulepath{flaml\_paper\_1h} constraint. The reported classification
evaluation covers 12 datasets. The regression benchmark was built from the
Penn Machine Learning Benchmarks (PMLB) repository, available at
\url{https://github.com/EpistasisLab/pmlb.git}, and was also executed through
AMLB. The reported regression evaluation covers 10 datasets and uses the
\modulepath{pmlb\_regression} suite with the \modulepath{pmlb\_30m}
constraint.

\noindent All systems were run through the same benchmark interface. The recorded setup
used one fold and one allocated core per task. The regression time budget was
1,800 seconds per dataset; the classification constraint allowed up to one
hour.

\chapter{Conclusions and Future Work}
\label{chap:conclusion}

This chapter summarizes the project outcomes and reflects on the experience
gained during development. It reviews the main challenges faced during the
implementation and evaluation of \systemname{}, states the conclusions reached
from the reported results, and identifies future extensions that can improve
the module.

\section{Faced Challenges}

The main challenges were balancing predictive quality with runtime, preserving
fair evaluation across different estimator families, isolating trial failures,
and keeping the search process inspectable. Multi-fidelity evaluation reduced
some search cost, but it required careful promotion rules so that cheap early
signals did not dominate final model selection unfairly.

\section{Gained Experience}

The project provided experience in AutoML system design, conditional search
spaces, Bayesian optimization, cross-validation protocols, process-level
timeouts, Python package resource handling, and ensemble construction. It also
showed the importance of benchmark reproducibility and of exposing run history
instead of returning only a final fitted model.

\section{Conclusions}

\systemname{} provides an Accelera-integrated AutoML module for supervised
tabular classification and regression. It combines conditional model spaces,
staged fidelity, surrogate-guided search, meta-learning warm starts, timeout
isolation, and voting or stacked ensembles behind a scikit-learn-style public
API. The stored experiments show predictive performance close to established
AutoML systems while reducing mean runtime in the recorded benchmark runs.

\section{Future Work}

Future work should include repeated benchmark folds, broader datasets, stronger
reporting of failed tasks, expanded support for categorical preprocessing,
additional model families, time-series validation modes, distributed execution,
and a completed public benchmark reproduction repository.

\subsection{ConfigSpace}

The \texttt{ConfigSpace} package is used for ConfigSpace-style conditional
configuration handling. In \systemname{}, it represents the AutoML search space
as a set of typed hyperparameters with activation conditions. A global
\texttt{model\_name} hyperparameter selects the active estimator family, while
model-specific preprocessing and hyperparameter choices are enabled only when
their parent model or option is active. This prevents invalid combinations from
being sampled and allows the optimizer, warm-start loader, and evaluator to work
with a single valid configuration object.

\chapter{Use Cases}

The main use cases of \systemname{} are:

\begin{itemize}
  \item fitting a classifier and obtaining predicted labels or class
        probabilities;
  \item fitting a regressor and obtaining numeric predictions;
  \item restricting the search to an allowed subset of model families;
  \item selecting the evaluation metric and the number of cross-validation
        folds;
  \item limiting the total search budget or the timeout for each individual
        trial;
  \item enabling or disabling meta-learning warm starts and ensemble
        construction;
  \item inspecting the best score, leaderboard, and complete run history.
\end{itemize}

\chapter{User Guide}

\systemname{} follows a scikit-learn-style workflow. The user first imports
either \texttt{AutoMLClassifier} or \texttt{AutoMLRegressor}, creates an
estimator with the required search settings, fits it on the training data, and
then uses the fitted model for prediction, scoring, or leaderboard inspection.

\noindent The following example shows the basic classification workflow:

\begin{lstlisting}[style=pythonstyle]
from accelera_automl import AutoMLClassifier

model = AutoMLClassifier(
    time_budget=1800,
    cv=5,
    scoring="accuracy",
    use_ensemble=True,
    random_state=42,
)

model.fit(X_train, y_train)

predictions = model.predict(X_test)
score = model.score(X_test, y_test)
leaderboard = model.return_leaderboard()
\end{lstlisting}

\noindent In this example, the search is limited to 1,800 seconds, five-fold
cross-validation is used, and accuracy is selected as the optimization metric.
After fitting, the model can generate predictions, evaluate performance on test
data, and return the stored leaderboard for inspection.

\chapter{Feasibility Study}

The proposed module is technically feasible because it is built on established
Python machine-learning libraries such as scikit-learn and can be integrated
within the existing Accelera package structure. The implementation does not
require specialized hardware and can run on standard CPU-based environments,
although larger search budgets and larger datasets require more computation
time.

\medskip
\noindent From an operational perspective, the module is feasible because it provides a
simple scikit-learn-style interface. Users can create an AutoML classifier or
regressor, call \texttt{fit}, and then use standard methods such as
\texttt{predict}, \texttt{score}, and \texttt{return\_leaderboard}. This makes
the system suitable for users who want automated model selection without
manually testing many algorithms and hyperparameter settings.

\medskip

\noindent The project is also economically feasible because it mainly depends on
open-source software tools and does not require paid cloud services or
commercial AutoML platforms. Reusing the same module across multiple
classification and regression tasks can reduce the manual effort needed for
model selection, tuning, and comparison.

\medskip
\noindent The main limitation is computational cost. AutoML search can be time-consuming,
especially when many models, folds, and trials are evaluated. However, this cost
is controlled through time budgets, per-trial timeouts, multi-fidelity
evaluation, and configurable search settings. Therefore, the project is feasible
as a practical AutoML extension for the Accelera framework.
